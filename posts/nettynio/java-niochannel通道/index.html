  1. 通道基础

    1. 从Channel接口引申出的其他接口都是面向字节的自接口，包括WritableByteChannel和ReadableByteChannel。通道只能在字节缓冲区上操作。

    2. SocketChannel sc = SocketChannel.open( );
sc.connect (new InetSocketAddress ("somehost", someport)); ServerSocketChannel ssc = ServerSocketChannel.open( ); ssc.socket( ).bind (new InetSocketAddress (somelocalport)); DatagramChannel dc = DatagramChannel.open( );
RandomAccessFile raf = new RandomAccessFile ("somefile", "r"); FileChannel fc = raf.getChannel( );

    3. RandomAccessFile.getChannel()方法只有在通道存在的时候才返回一个socket关联的通道。它们永远不会创建新通道。

    4. 使用通道

      1. ByteChannel接口

      2. 通道可以以阻塞或非阻塞模式运行。非阻塞模式的通道永远不会让调用的线程休眠。请求的操作要么立即完成，要么返回一个结果表明未进行任何操作。只有面向流的（stream-oriented）的通道，如 sockets 和 pipes 才能使用非阻塞模式。

    5. 关闭通道

      1. isOpen()：

        1. 如果返回true，表示该通道可以使用，如果返回false，表示该通道已关闭，不能再被使用
        2. 在一个通道多次调用close()方法是没有坏处的，但是如果第一个线程在close()方法中阻塞，那么在它关闭通道之前，其它任何调用close()方法都会阻塞。在已关闭的通道上调用close()不会产生任何操作，只会立即返回。

    6. Scatter/Gather

      1. 它是在多个缓冲区实现一个简单的I/O操作，对于一个write操作而言，数据是从几个缓冲区按顺序抽取(gather)并沿着通道发送的。对于read操作而言，从通道读取的数据会按顺序被散布(scatter)到多个缓冲区

      2. 接口

      3. offset，length

        1. offset表示哪个缓冲区将开始被使用，length表示要使用的缓冲区数量

    7. 文件通道

      1. 文件通道总是阻塞式的，FileChannel对象是线程安全的。不过影响通道位置或者影响文件大小的操作都是单线程的

      2. 访问文件

        1. public abstract long position():返回当前文件的position值，返回值是一个长整型，表示文件中的当前字节位置
        2. public abstract void position(long newPosition)：将通道的position设置为指定值
          1. position不能为负值，如果position设置了超出文件尾，这样会把position设为指定值而不改变文件大小。如果此时实现了read方法，会返回（end-of-file），如果此时实现的是一个write()方法，则会引起文件增长以容纳写入的字节，可能导致出现文件空洞
        3. 当字节被read()或write()时，文件position会自动更新。如果position达到了文件大小的值，read()方法会返回文件尾条件值(-1)，如果实现write方法()时position前进到超过文件大小的值，该文件会扩展以容纳新写入的字节
        4. truncate()：会砍掉所指定的新size值之外的所有数据，文件的position会被设置为所提供的新size值
        5. force(boolean metaData):要求文件的所有待定修改立即同步到磁盘
          1. 如果文件位于一个本地文件系统，那么一旦修改force()方法返回，即可保证从通道被创建时对文件所做的修改已经被写入到磁盘，可以保证数据完整性和可靠的恢复。如果文件位于一个远程的文件系统，那么不能保证待定修改一定能同步到永久存储器上。
          2. 方法的布尔型参数表示在方法返回值前文件的元数据是否也要被同步更新到磁盘。元数据指文件所有者，访问权限，最后一次修改时间等信息。给force()传递false值表示在方法返回前只需要同步文件数据的更改。

      3. 文件锁定

        1. public abstract FileLock lock(long position, long size, boolean shared):

          1. 会指定文件内部锁定区域的开始position以及锁定区域的size，第三个参数shared表示向获取的锁是共享的(true)还是独占的(false),

        2. lock():等价于lock(0L, Long.MAX_VALUE, false)

        3. tryLock()：和lock()方法有一样的作用，不过如果请求的锁不能立即获取到则会返回一个null

        4. FileLock对象：

          1. lock()和tryLock()均返回一个FileLock对象，
          2. channel()：查询一个lock对象以判断它是由哪个通道创建的
          3. 一个FileLock对象创建之后即有效，知道它的release()方法被调用才会失效，可以通过调用isValid来测试一个锁的有效性。一个锁的有效性可能会随着时间而改变，不过它的其他属性——position，size和独占性(exclusivity)不会随着时间而改变。
          4. isShared():判断一个锁是独占还是共享的
          5. overlaps()：查询一个FIleLock对象是否与一个指定的文件区域重叠

      4. 内存映射文件

        1. public abstract MappedByteBuffer map(MapMode mapMode, long position, long size)：通过FileChannel的map()方法来创建一个文件映射，它的参数有MapMode，position，size.我们可以创建一个MapppedByteBuffer来代表一个文件中字节的某个子范围。映射范围不能超过文件的实际大小

        2. 文件的映射可以是可写的或只读的。请求的映射模式将受被调用map()方法的FileChannel对象的访问权限所限制。

        3. MapMode.PRIVATE表示您想要一个写时拷贝的映射，这意味着通过put()方法所做的任何修改都会导致一个私有的数据拷贝并且该拷贝中的数据只有MappedByteBuffer实例可以看到，任何修改不会回刷到文件中

        4. load()：加载整个文件以使它常驻内存

        5. isLoaded():判断一个被映射的文件是否完全常驻内存了。如果该方法返回 true 值，那么很大概率是映射缓冲区的访问延迟很少或者根本没有延迟。不过，这也是不能保证的。同样地，返回 false 值并不一定意味着访问缓冲区将很慢或者该文件并未完全常驻内存。

        6. force()：强制将映射缓冲区上的更改应用到永久磁盘存储器上。如果映射是以 MapMode.READ_ONLY 或 MAP_MODE.PRIVATE 模式建立的，那么调用 force( ) 方法将不起任何作用，因为永远不会有更改需要应用到磁盘上（但是这样做也是没有害处的）。

        7. Channel-to-Channel传输

          1. transferTo(long position, long count, WritableByteChannel target), transferFrom(ReadableByteBuffer src, long position, long count):transferTo()和transferFrom()方法允许一个通道交叉连接到另一个通道，只有FileChannel类有这两个方法，因此channel-to-channel传输中通道之一必须是FileChannel。

    8. Socket通道

      socket通道类分为DatagramChannel，SocketChannel和ServerSocketChannel三种，其中DatagramChannel和SocketChannel实现定义读和写功能的接口而ServerSocketChannel不实现。ServerSocketChannel负责监听传入的连接和创建新的SocketChannel对象，它本身从不传输数据。

      1. 非阻塞模式

        1. 要把一个socket通道置于阻塞模式，需要依靠所有socket通道类的共有超级类：SelectableChannel。

          1. public abstract void configureBlocking(boolean block)：传递参数true则设为阻塞模式

          2. public sbatract boolean isBlocking

          3. public abstract Object blockingLock()：

            1. 为了防止socket通道的阻塞模式被修改，该方法返回一个非透明的对象引用。返回的对象是通过实现修改阻塞模式时内部使用的，只有拥有此对象的锁的线程才能更改通道的阻塞模式。

          4. 非阻塞I/O和可选择性是紧密关联的

      2. ServerSocketChannel：

        1. open():创建一个新的ServerSocketChannel对象，将会返回一个未绑定的java.net.ServerSocket关联的通道
        2. socket()：取出相应的ServerSocket并使用它来绑定到一个端口以开始监听连接
        3. accept()：如果选择在ServerSocket上调用accept()方法，它总是阻塞并返回一个java.net.Socket对象，如果选择在ServerSocketChannel上调用accept()方法则会返回SocketChannel类型的对象，返回的对象能够在非阻塞模式下运行。

      3. SocketChannel

        每个SocketChannel对象创建时都是同一个对等的java.net.Socket独享串联的。静态的open()方法可以创建一个新的SocketChannel对象，而在新创建的SocketChannel上调用socket()方法能返回它对等的Socket对象；在该Socket上调用getChannel()方法能返回最初的那个SocketChannel

        1. 假如在一个非阻塞模式的 SocketChannel 对象上调用 finishConnect( *)*方法，将可能出现下列情形之一：
          1. connect( *)*方法尚未被调用。那么将产生 NoConnectionPendingException 异常。
          2. 连接建立过程正在进行，尚未完成。那么什么都不会发生，finishConnect( *)*方法会立即返回false值
          3. 在非阻塞模式下调用 connect( *)*方法之后，SocketChannel 又被切换回了阻塞模式。那么如果有必要的话，调用线程会阻ຎ直到连接建立完成，finishConnect( *)*方法接着就会返回 true值
          4. 在初次调用 connect( *)*或最后一次调用 finishConnect( )之后，连接建立过程已经完成。那么SocketChannel 对象的内部状态将被更新到已连接状态，finishConnect( *)*方法会返回 true值，然后 SocketChannel 对象就可以被用来传输数据了。
          5. 连接已经建立。那么什么都不会发生，finishConnect( *)*方法会返回 true 值。

      4. DatagramChannel

        DatagramChannel模拟包导向的无连接协议(UDP/IP)

        1. open()
          1. 调用静态的open()方法来创建DatagramChannel实例

        2. 数据的实际发送和接收是通过send()和receive()方法来实现的

        3. public abstract SocketAddress receive(ByteBuffer dst):

          1. receive( *)*方法将下次将传入的数据报的数据复制到预备好的 ByteBuffer 中并返回一个 SocketAddress 对象以指出数据来源。
          2. 如果通道处于阻塞模式，receive( *)*可能无限期地休眠直到有包 到达。如果是非阻塞模式，当没有可接收的包时则会返回 null。
          3. 如果包内的数据超出缓冲区能承受的范围，多出的数据都会被悄悄地抛弃。

        4. public abstract int send (ByteBuffer src, SocketAddress target):

          1. 调用 send( *)*会发送给定 ByteBuffer 对象的内容到给定 SocketAddress 对象所描述的目的地址和端 口，内容范围为从当前 position 开始到末尾处束。
          2. 如果 DatagramChannel 对象处于阻塞模式，调用线程可能会休眠直到数据报被加入传输队列。
          3. 如果通道是非阻塞的，返回值要么是字节缓冲区的 字节数，要么是“0”。发送数据报是一个全有或全无（all-or-nothing）的行为。如果传输队列没有足够空间来承载整个数据报，那么什么内容都不会被发送。
